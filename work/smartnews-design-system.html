SmartNews · 2024

SmartNews Design System Foundations

Company SmartNews, Inc.
Timeline Aug – Nov 2024 (4 months)
Role Senior Product Designer (Sole Designer)
🎨
"Design system quality lies in the invisible."

Overview

SmartNews had grown rapidly — but its design foundation hadn't kept pace. Inconsistent UI patterns, duplicated components, and misaligned color tokens were slowing down the team. As the sole designer on this initiative, I was tasked with building SmartNews' first formal design system from scratch.

The goal: establish a shared design language that designers and engineers could build with confidence, and reduce the friction caused by inconsistencies across the product.

The Problem

Before the design system, the team faced several compounding issues:

  • No consistent naming conventions between design and engineering
  • Color tokens undefined — every designer used slightly different values
  • No formal light/dark mode token strategy
  • Components were duplicated across files with inconsistent variants
  • Designer–engineer handoff was slow due to unclear specifications

The Approach

Color Primitives → Color Tokens

I started at the foundation: color. The first step was establishing color primitives (the raw palette) and mapping them to semantic color tokens (what each color means in context). Light and dark mode were addressed simultaneously from day one, not retrofitted later.

Token naming was a key focus. Rather than technical names like gray-700, tokens used semantic names like text-primary or surface-default — vocabulary that meant the same thing to a designer and an engineer.

Typography System

I standardized type scales, weights, and line heights into a clear hierarchy. Each style was given a clear role: heading, subheading, body, caption, label — with specific size and weight pairings that scaled predictably across breakpoints.

Component Library

Working from the most-used components first, I audited the existing UI, identified patterns, and rebuilt them as proper Figma components with auto-layout, variants, and interactive states.

What I Learned

Building a design system solo is humbling. You quickly realize that a design system is never "done" — it's a living document that must grow with the product. The most valuable thing I built wasn't the components themselves, but the naming frameworks and decision rules that helped the whole team understand why things were the way they were.

The biggest win? When an engineer used the token text-primary in their PR and a designer immediately understood what it meant without a handoff doc. That shared language was the real product.
